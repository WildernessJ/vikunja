## Findings (ranked)

### F1 — Assignee autocomplete ignores the typed `+project` (CORRECTNESS) — act
`AddTask.vue` `assigneeProjectId` computed resolves only the project *chip override* or route/default —
it never reads the parsed `+project` name from the title (`effectiveProjectName`). So typing
`+Work @al` scopes the assignee suggestion search to the route/default project, not "Work" → wrong/no
assignee suggestions for a common workflow. The created task still resolves correctly (createNewTask
uses `findProjectId`, which tries the parsed name first) — this is an autocomplete-scoping bug only.
Fix is cheap + sync: resolve `effectiveProjectName` via `projectStore.findProjectByExactname(name)?.id`
before the route/default fallback. **Decision: fix before merge.**

### F2 — No e2e for reminders chip + Popup-inert (COMPLETENESS) — accept/defer
Scopes 3 and 5 shipped without e2e or a recorded browser session. Unit-covered; scope 2 was
live-verified by Jason. Reminders chip mirrors verified chips; Popup-inert is verified structurally.
**Decision: fold into the final combined live-verify; no e2e added (consistent with the composer's
existing test strategy).**

### F3 — Popup `inert` is a latent focus-on-open landmine (A11Y/maintainability) — note
No current `<Popup>` consumer auto-focuses its content on open (all 7 checked), so no live bug. But
content mounts once while `inert`, so a *future* `v-focus` inside popup content would silently fail.
**Decision: add a one-line why-comment on Popup.vue documenting this; no behavioral change.**

### F4 — `tokenAtCaret` truncates on an apostrophe inside a quoted name (INHERITED) — defer
`tokenAtCaret("+'Bob's Project'", 16, VIK)` → null. Mirrors a pre-existing `getItemsFromPrefix` bug
(untouched here); surfaced more visibly by live autocomplete. Not a new regression.
**Decision: follow-up note, no fix this branch.**
